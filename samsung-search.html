Hybrid AI-powered Product Search

Improving on-site search relevance by selectively augmenting lexical retrieval with semantic understanding.

Domain
E-commerce · Search · AI/ML
Tech
Vector embeddings · Semantic retrieval · Hybrid ranking
Overview

Improve intent-driven search while preserving exact-match precision.

Samsung.com site search relied primarily on keyword-based lexical matching, which worked well for exact product queries but struggled with ambiguous, long-tail, and natural-language shopping intent. We selectively introduced semantic retrieval to improve recall for intent-driven searches while preserving lexical search for high-precision exact matches. As the PM, I defined the solution strategy, retrieval quality metrics, and phased rollout plan, delivering a 30% lift in Precision@3 and a 45% increase in first-result CTR.

The Problem

Keyword-based search failed on ambiguous and long-tail queries.

Samsung.com's on-site search was built primarily on keyword matching logic (lexical search). That worked well for exact queries, but failed when intent was ambiguous. A search like "Galaxy Watch" could return a mix of phones, watches, and TVs because the engine matched shared tokens (keywords) rather than actual shopping intent.

The short-term fix was to layer on rules: category dictionaries, synonym maps, homonym handling, and redirect logic. Each rule addressed a specific failure case, but the system became increasingly brittle. Fixing one query could create regressions elsewhere, and surfacing the right products meant managing an exception list that grew faster than the team could keep up with.

The Solution

A hybrid system: semantic retrieval where keywords fail, keyword matching where they still work.

AI could improve relevance for ambiguous queries, but introducing it raised two questions: where to apply it, and how to integrate it. The solution was a hybrid AI search system: a semantic layer was added for ambiguous queries where keyword matching was failing, while the existing lexical pipeline continued to handle exact-match queries.

Two decisions shaped this approach.

01
Scope: target failing queries first
Rather than applying AI universally, I focused on queries where keyword-based search was failing. To identify them, I used low search-result CTR and high no-result rates as signals. Queries that didn't lead to clicks, whether from missing or irrelevant results, in aggregate represented a meaningful share of failed searches even at low individual volume.
02
Architecture: selective augmentation, not full replacement
Keyword-matching logic still performed well for exact-match queries like "Galaxy S24 Ultra" where precision mattered most. A full replacement would have introduced unnecessary risk, so I kept the lexical search and applied semantic retrieval only where it was failing. I validated this with a PoC using semantic embeddings, then designed a routing layer that preserved the lexical pipeline for exact-match queries while sending ambiguous and low-engagement queries to vector retrieval. The lexical search was retained as a fallback so search could continue if the AI engine failed. The model was released progressively across eligible query traffic, with query-level evaluation and engagement metrics monitored at each stage.
Trade-offs

Three tensions, and how I resolved each.

Semantic recall vs. exact-match precision. Semantic retrieval improved ambiguous queries, but applying it universally risked over-generalizing exact product searches and returning related but unintended results.

Generalization vs. explainability. Rule-based systems were brittle but easy to trace. Vector retrieval handled new query patterns without hand-coded rules, but the cost was explainability. When results were wrong, it was harder to diagnose the cause. This was a concern for business and category management teams, who needed to understand why specific products surfaced. I treated this as a rollout risk and managed it through phased rollout, query-level evaluation, and metric monitoring so regressions could be caught before full roll-out.

Initial complexity vs. long-term scalability. Adding a semantic layer introduced upfront complexity, but continuing to patch the rule system meant maintaining an exception list with no ceiling. The added complexity was a worthwhile cost to avoid accumulating ops and tech debt indefinitely.

Impact

Search quality improvements that customers responded to.

The rollout was evaluated across two layers: retrieval quality (Precision@3, MRR, Recall) and user behavior (search CTR, search-assisted conversion, query reformulation rate). This combination ensured improvements in semantic relevance translated into actual shopping behavior change.
