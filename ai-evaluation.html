A Practical Framework for AI Evals

AI products are probabilistic. The same input can yield different outputs, so evaluating them requires a more structured and layered approach than traditional products.

Most traditional products are deterministic. When the same input is given, the system produces the same output based on predefined logic. Success can be measured through clearly defined user flows and metrics.

AI products, however, are inherently probabilistic. The same input can yield different outputs. While we can tune parameters like temperature or tone, the output is not guaranteed to be consistent. Because of this, evaluating AI products requires a more structured and layered approach.

This post outlines a practical, end-to-end framework for evaluating AI-powered products with a real-world example.

0. Define What Success Means

As always, the first step is to clearly define what success looks like.

For AI products, this should be defined at two levels, for both the user and the product. I recommend starting from user-level success and then breaking it down into product-level metrics.

User-level success

  • Did the user accomplish their goal?
  • Did the system reduce effort or time?

Product-level success

  • Task success rate
  • Time to resolution
  • User satisfaction

For example, in a conversational product discovery experience, success is about helping users find the right products or information more easily and quickly, so they can make confident purchase decisions.

Without this step, evaluation becomes disconnected from actual user value and risks over-optimizing proxy metrics.

1. Establish a Golden Dataset

Once success is defined, the next step is to create a reliable evaluation baseline, a golden dataset.

A golden dataset consists of input-output pairs. Inputs are real user queries (including long-tail and ambiguous cases), and ideal outputs are high-quality responses defined with domain experts.

In a brand chatbot context, this step involves categorizing use cases by user intent, such as product discovery, comparison, and support-related queries, and defining corresponding answer sets. In practice, I worked closely with domain experts (e.g., brand managers, marketing teams) to define what good responses look like across these use cases.

This dataset provides a consistent standard for measuring system quality and enables repeatable evaluation.

2. Offline Evaluation (System Quality)

Offline evaluation measures how well the system performs in a controlled environment before production. Typical evaluation approaches include the following:

  • Rubric-based scoring (manual or model-assisted)
  • Accuracy and factuality checks
  • Hallucination detection
  • Coverage of edge cases

For example, in a product-focused chatbot, offline evaluation can measure whether responses correctly recommend relevant products and provide sufficient supporting details (e.g., key features, differences).

At this stage, the key question is "Can the system produce high-quality outputs?"

In this stage, two complementary approaches help scale offline evaluation.

First, LLM-as-judge uses a separate language model to score responses against a rubric. This enables fast and repeatable evaluation across large response sets.

Second, human-in-the-loop review runs in parallel. Domain experts periodically validate a sample of the golden dataset to ensure the rubric remains reliable and to catch blind spots that automated scoring may overlook.

3. Online Experimentation (Controlled Comparison)

Once baseline quality is established, it's time to test it in a live but controlled environment through experimentation.

This stage focuses on comparing variants, not measuring final success.

Typical experiments include:

  • Different prompting strategies
  • Retrieval configurations (e.g., number or type of documents)
  • Response formats (e.g., concise vs. structured with explanations)

These experiments test system-level variations that run on top of the same underlying model (rather than testing different models themselves).

For example, one variant might use a concise response with minimal context, while another uses a more structured format with richer context and citations. In one case, I tested these variations to understand which better supports user decision-making during product discovery.

Metrics at this stage are comparative signals, such as:

  • Click-through rate (CTR)
  • Re-try rate
  • Conversation completion

The key question here is, "Which version performs better relative to another?"

This stage helps identify better configurations. However, it does not yet confirm whether the product is truly effective.

4. Behavioral & Outcome Metrics (Real Impact)

Now, after selecting the strongest variant, it's time to measure actual user impacts.

This stage answers a fundamentally different question, "Did the product actually help the user achieve their goal?"

Unlike online experimentation, this is not about comparing versions, but about validating real impact and measuring success based on what we defined from the start.

Key metrics include:

  • Task completion rate
  • Reduction in follow-up queries
  • Conversion or downstream actions
  • User satisfaction

For example, even if a chatbot increases CTR or engagement, a high rate of follow-up questions may indicate that responses are not fully resolving user needs.

5. Continuous Feedback Loop

Like any other product, AI products require continuous iteration.

Feedback signals include the following:

  • Explicit product feedback (ratings, thumbs up/down)
  • Implicit behavioral signals (query reformulation, abandonment, repeated interactions)

These signals can be used to continuously improve the system by:

  • Improving prompts or retrieval strategies
  • Refining ranking logic
  • Updating training data

Evaluate → Learn → Improve → Re-evaluate.

This loop is what enables AI products to continuously improve and stay aligned with real user needs.

Evaluation Principles: Helpful, Honest, Harmless

Across all of these stages, there's one framework that I keep in the back of my mind: the "HHH" (Helpful, Honest, and Harmless) framework from Anthropic. This is a useful northstar for setting quality criteria.

In my experience, this framework is especially useful for translating abstract evaluation goals into concrete criteria:

  • Helpful. Does the response actually solve the user's problem? Measured through task completion and relevance.
  • Honest. Does the product avoid making up information when uncertain? Does it clearly acknowledge when information is unavailable? Evaluated through hallucination checks and fallback behavior.
  • Harmless. Does the system avoid unsafe guidance or inappropriate responses? Enforced through policy constraints and safety evaluation.

In practice, I always think about whether the system is hitting all three at the same time. Embedding these principles into evaluation criteria ensures the system is not only effective, but also trustworthy.

Key Takeaways

Evaluating AI products is not a one-time process. It requires a layered framework that connects model performance to real user outcomes by following these steps:

01
Define success based on user goals
02
Build a golden dataset
03
Evaluate system quality offline
04
Compare variants through experimentation
05
Measure real-world outcomes
06
Continuously improve through feedback

Ultimately, strong AI products are not defined by how well they generate responses, but by how effectively they help users accomplish what they came for.
